# План развития music-dl

Дорожная карта: CLI → десктоп (Windows + macOS) → сайт → бэкенд (по мере необходимости) → мобильные приложения (Android + iOS). Провайдер VK Music и расширяемая архитектура под новые сервисы.

---

## Текущее состояние (v2.0)

- CLI + обёртки `ym.ps1` / `ym.cmd`
- Провайдер **Яндекс.Музыка** (`music_dl/providers/yandex/`)
- Модульная архитектура: `core` → `providers` → `pipeline` → `cli`
- Репозиторий: [Evgeniy16312/music-dl](https://github.com/Evgeniy16312/music-dl)

## Фаза 3 — Десктоп-приложение с UI

**Цель:** Win + macOS, без терминала для обычных пользователей.

### 3.1 Выбор стека

| Компонент | Рекомендация |
|-----------|--------------|
| UI | **Flet** (Python, Win/Mac, современный вид) |
| Бэкенд | существующий `music_dl` (без дублирования логики) |
| Сборка Win | **PyInstaller** → `.exe` |
| Сборка Mac | **PyInstaller** на macOS или GitHub Actions → `.app` / DMG |

Альтернатива UI: CustomTkinter (проще, менее «app-like»).

### 3.2 Структура

```
music_dl/
  gui/
    app.py           # точка входа Flet
    views/
      home.py        # URL + скачать
      auth.py        # вход Yandex / VK
      settings.py    # папка, качество, workers
      progress.py    # прогресс, лог, стоп
    bridge.py        # вызов pipeline в фоновом потоке
```

- [ ] Скачивание в **отдельном потоке** — UI не блокируется
- [ ] Callback прогресса из `pipeline/runner.py` (сейчас Rich → GUI events)
- [ ] Ctrl+C заменить кнопкой «Стоп»

### 3.3 Экраны (MVP)

1. **Главная** — поле URL, «Вставить из буфера», выбор провайдера (авто), «Скачать»
2. **Авторизация** — вкладки Yandex / VK, статус «вошёл / нет»
3. **Настройки** — папка, MP3/AAC, битрейт, workers
4. **Прогресс** — список треков, общий %, кнопка «Остановить»

### 3.4 Упаковка

- [ ] `pyproject.toml` — optional `[gui]` dependencies: `flet`
- [ ] `build/windows.spec` / `build/macos.spec` для PyInstaller
- [ ] GitHub Actions: сборка артефактов на push tag `v*`
- [ ] Releases: `.exe` (Win), `.dmg` или `.zip` (Mac)

### 3.5 macOS нюансы

- Подпись кода (Apple Developer) — для комфортной установки у других
- Без подписи — инструкция «ПКМ → Открыть» в README

**Оценка MVP GUI:** 2–3 недели.  
**Оценка сборки + CI:** 3–5 дней.

## Фаза 6 — Бэкенд

**Короткий ответ:** для CLI и десктопа **бэкенд не нужен** — приложение ходит напрямую в API Yandex/VK, токены хранятся локально. Бэкенд понадобится, когда появятся **сайт с аккаунтами**, **мобильные приложения в магазинах** или **безопасный OAuth без секретов в клиенте**.

### 6.1 Когда бэкенд нужен, а когда нет

| Сценарий | Бэкенд |
|----------|--------|
| CLI / десктоп, токен в `config.json` | **Не нужен** |
| Статический сайт + GitHub Releases | **Не нужен** |
| OAuth VK/Yandex в мобильном app (client_secret нельзя в APK) | **Нужен** — прокси авторизации |
| Проверка версии / changelog JSON | **Опционально** — хватит статического JSON на CDN |
| Облачная синхронизация настроек между устройствами | **Нужен** |
| Веб-версия «вставь URL — скачай» | **Нужен** (и высокий юридический риск) |
| Push-уведомления «загрузка завершена» | **Нужен** (FCM/APNs через сервер) |

**Рекомендация:** не строить бэкенд «на будущее». Ввести **минимальный API** перед мобильным релизом (Фаза 7) или когда OAuth на телефоне станет блокером.

### 6.2 Архитектура (целевая)

```
Клиенты                    Бэкенд                         Внешние API
────────                   ───────                        ───────────
Desktop (Flet)  ──┐
Mobile (Flet)   ──┼──►  music-dl-api  ──►  OAuth proxy  ──►  VK / Yandex
Website (opt.)  ──┘         │              (токены)
                            │
                       PostgreSQL
                       (users, sessions)
                            │
                       Redis (опц.)
                       (очереди, кэш)
```

**Важно:** логика скачивания и провайдеры остаются в **`music_dl` (Python)**. Бэкенд не дублирует pipeline — он занимается **auth, метаданными пользователя, версиями, аналитикой**. Скачивание по-прежнему на устройстве пользователя.

### 6.3 Стек бэкенда (рекомендация)

| Компонент | Выбор | Почему |
|-----------|-------|--------|
| API | **FastAPI** (Python) | Тот же язык, можно импортировать части `music_dl` |
| БД | **PostgreSQL** (prod), SQLite (dev) | Пользователи, refresh-токены |
| Auth | OAuth2 + JWT | VK / Yandex callback на сервер |
| Деплой | Railway, Fly.io, VPS, Timeweb | Небольшой трафик на старте |
| Секреты | env / vault | client_secret только на сервере |

### 6.4 MVP API (первая итерация)

```
GET  /health
GET  /v1/version              # последняя версия клиента, URL релиза
GET  /v1/auth/{provider}/url  # redirect URL для OAuth
GET  /v1/auth/{provider}/callback
POST /v1/auth/refresh
GET  /v1/me/providers         # какие сервисы подключены (без хранения музыки на сервере)
```

- [ ] Регистрация приложений VK / Yandex с redirect на `api.music-dl.example/oauth/callback`
- [ ] Хранение **refresh token** зашифрованно; access token — короткий TTL
- [ ] Rate limiting, CORS только для своих клиентов
- [ ] **Не** проксировать сами аудиофайлы через сервер (нагрузка + правовые риски)

### 6.5 Структура кода

```
backend/                    # отдельная папка или repo music-dl-api
  app/
    main.py
    routers/
      auth.py
      version.py
    services/
      oauth_vk.py
      oauth_yandex.py
    models/
    db/
  requirements.txt
  Dockerfile
```

**Оценка MVP бэкенда:** 1–2 недели (OAuth + version endpoint).  
**Оценка с аккаунтами и sync:** +2–3 недели.

## Фаза 7 — Мобильные приложения (Android + iOS)

**Цель:** тот же функционал на телефоне — URL / Share → скачать в память устройства.

### 7.1 Подход

| Компонент | Рекомендация |
|-----------|--------------|
| UI | **Flet mobile** (переиспользование `music_dl/gui/` с адаптацией) |
| Логика | `music_dl` — providers + pipeline (как на десктопе) |
| Auth | через **бэкенд OAuth** (Фаза 6) или in-app browser + deep link |
| Android | APK / AAB → Google Play или прямая раздача APK с сайта |
| iOS | TestFlight → App Store (нужен Apple Developer, $99/год) |

PyInstaller и `.exe` **не переносятся** на телефон — отдельная сборка Flet (`flet build apk` / `ipa`).

### 7.2 Отличия от десктопа

- [ ] UI под маленький экран: bottom navigation, жесты
- [ ] **Share intent** / **Share Extension** — «Поделиться» URL из VK / Яндекс.Мож
- [ ] Папка сохранения: Music / Downloads + разрешения Android
- [ ] **Foreground service** (Android) для длинных загрузок
- [ ] iOS: ограничения фоновой работы — уведомление «не закрывай приложение»
- [ ] Deep link: `musicdl://download?url=...`

### 7.3 Зависимость от бэкенда

```
Десктоп MVP     →  без бэкенда ✅
Mobile MVP      →  желательно минимальный бэкенд для OAuth 🔶
Mobile + магазины →  бэкенд для OAuth почти обязателен ✅
```

### 7.4 Риски магазинов

Google Play и App Store могут **отклонить** приложение для скачивания музыки из стриминг-сервисов. Варианты:

- Публикация APK **только с сайта** (Android)
- TestFlight / ограниченная аудитория (iOS)
- Чёткий disclaimer: личное архивирование

**Оценка Android MVP:** 2–4 недели после готового десктопа GUI.  
**Оценка iOS:** +2–3 недели (сертификаты, тесты на устройствах).

## Риски

| Риск | Митигация |
|------|-----------|
| VK API меняется / закрывается | Абстракция провайдера, версионирование |
| Блокировка неофициального API | Документировать лимиты, только личное использование |
| Большой размер .exe (~100 MB) | Норма для Python; UPX / one-folder build |
| Mac без подписи | Инструкция в README + позже подпись |
| Отклонение в App Store / Play | APK с сайта, disclaimer, без публичного web-download |
| Преждевременный бэкенд | Начинать с OAuth-only MVP, не дублировать pipeline на сервере |
| Стоимость инфраструктуры | Статика бесплатно; API — от ~$5–20/мес на малом VPS |
